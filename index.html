《持续交付--发布可靠软件的系统方法》读书心得

发表于 2017-10-01 | 分类于 读书心得

核心还是自动化！

敏捷里面有一个名言，让你痛苦的事情，提前、频繁的做。（If it hurts you，do if offen.）
但是要做到这些，必须有自动化的基础，之后实行自动化，才能重复的、可靠的去做。 包括自动单元测试、自动构建、自动验收测试、自动部署、自动发布等。
要实现自动化，测试的自动化，需要开发、测试的通力合作，编写能自动运行的测试脚本；构建、部署的自动化，需要运维、开发的通力合作，来编写相关的自动构建和部署脚本、自动化的环境准备（虚拟化技术）、准备有版本控制的配置信息，这样能让所有的人员简单的、可重复的部署到各种环境，特别说明的是，自动化的环境需要包括对应的数据库模式初始化以及相关的数据加载，这些都必须通过脚本来处理。
这些，都是敏捷的一个核心原则，强调合作，也是后来DevOps的一种精神。

这些过程，其实都离不开脚本编写能力，所以自动化测试对测试工程师的要求就是有一定的编程能力。自动化运维也同理，虽然有很多工具的支撑，但是还是需要有一定的脚本编写能力才能更好的发挥相关工具的作用。

以上依赖于主干开发模式。

实例和数据库的关系

发表于 2017-09-26 | 分类于 技术相关

一个实例在其生存期中最多只能装载和打开一个数据库。要想再打开这个（或其他）数据库，必须先丢弃这个实例，并创建一个新的实例。 重申一遍：实例是一组后台进程和共享内存。 数据库是磁盘上存储的数据集合。 实例“一生”只能装载并打开一个数据库。 数据库可以由一个或多个实例（使用 RAC 数据库服务集群）装载和打开。 前面提到过，大多数情况下，实例和数据库之间存在一种一对一的关系。可能正因如此，才导致人们很容易将二者混淆。从大多数人的经验看来，数据库就是实例，实例就是数据库。

数据库版本管理

发表于 2017-09-26 | 分类于 技术相关

Active Record - 数据库迁移(Migration)
Programming today is a race between software engineers striving to build bigger and better idiot-proof programs, and the Universe trying to produce bigger and better idiots. So far, the Universe is winning. - Rick Cook
Migrations（数据库迁移）可以让你用 Ruby 程式来修改数据库结构。相较于直接进数据库系统使用 SQL 修改结构(例如使用 phpMyAdmin 工具来修改)，使用 Migrations 可以让我们有记录地进行数据库修改，每次变更就是一笔 Migration 记录。在没有 Migration 之前，如果你手动修改了数据库，那么你就必须通知其他开发者也进行一样的修改步骤。另外，在正式布署的服务器上，你也必须追踪并执行同样的变更才行。而这些步骤如果没有记录下来，就很容易出错。

区分workflow和BPM流的一个重点是看核心单据在流程中有无转化？如果全流程都是针对一个核心业务单据，一般是Workflow，而全流程出现多种不同的业务单据的流转，如从项目，到合同，再到订单等，一般就可以称得上是BPM了。
